# Coding Bot 使用手册

这份手册面向已经完成部署的使用者，重点说明：

- 机器人怎么触发
- 任务怎么流转
- 平时怎么看状态
- 任务失败后怎么处理

部署步骤请看 [部署.md](部署.md)。

## 1. 先理解它在做什么

`Coding Bot` 会监听你允许处理的 GitHub 仓库 Issue，然后在满足触发条件时自动执行下面这条链路：

1. 发现触发条件
2. 写入本地任务队列
3. 复用对应仓库的本地缓存目录
4. 创建工作分支并调用本地 Codex
5. 默认由 Codex 做最小自检；只有显式配置 `TEST_COMMAND` 时才额外执行外层测试命令
6. 推送到 fork
7. 创建 PR
8. 在 Issue 下回帖结果
9. 可选自动追加 `/submit`

默认推荐是 `polling-only` 模式，所以它不是收到评论瞬间执行，而是按轮询周期扫描。

## 2. 哪些仓库会被处理

只有 `.env` 里 `ALLOWED_REPOS` 配置过的仓库会被处理。

例如：

```env
ALLOWED_REPOS=yeying-community/deployer,yeying-community/robot
```

不在这个列表里的仓库，机器人会直接忽略。

## 3. 哪些操作会触发任务

触发方式由 `.env` 决定，当前支持三种：

- 评论触发：默认是评论 `/run`
- 标签触发：默认是打上 `ai-run`
- Issue 新建触发：只有 `RUN_ON_ISSUE_OPENED=true` 才开启

默认配置通常是：

```env
RUN_ON_ISSUE_OPENED=false
TRIGGER_LABEL=ai-run
TRIGGER_COMMENT=/run
```

如果你走的是默认轮询模式，还要看：

```env
ENABLE_POLLING=true
POLL_INTERVAL_SECONDS=60
```

这表示新评论 `/run` 一般会在下一次轮询时被发现，不是实时秒级触发。

## 4. 正常使用方式

最常见的用法是：

1. 在允许处理的仓库里新建或打开一个 Issue
2. 把需求写清楚
3. 评论 `/run`
4. 等待机器人处理

处理成功后，机器人通常会在 Issue 下回复：

- Job ID
- 处理结果
- PR 地址

如果 `.env` 里开启了：

```env
SUBMIT_COMMENT_AFTER_PR=true
SUBMIT_COMMENT_BODY=/submit
```

那它在创建 PR 后还会自动再评论一次 `/submit`。

默认建议把：

```env
TEST_COMMAND=
```

保持为空。这样机器人只依赖 Codex 在任务内做最小必要自检，不会再额外执行一条全局固定的 shell 测试命令。

## 5. 它怎么排队

当前排队规则是：

- 同一个仓库内串行处理
- 不同仓库之间可以各自排队
- 同一个 Issue 在已有 `queued` 或 `running` 任务时，新的触发会去重，不会同时跑多个活跃任务

这意味着：

- `repo A` 和 `repo B` 可以各跑一个任务
- `repo A` 里不会同时并发跑两个任务
- 同一个 Issue 在执行中时，再评论一次 `/run`，通常不会立刻再创建第二个活跃任务

如果前一个任务已经结束，你再评论一次新的 `/run`，会生成新的任务。

## 6. 怎么看机器人是不是活着

先看健康检查：

```bash
curl -s http://127.0.0.1:9081/health
```

重点看这些字段：

- `status`
- `allowed_repos`
- `poll_enabled`
- `last_poll_started_at`
- `last_poll_completed_at`
- `last_poll_error`
- `last_queued_job_id`
- `last_dispatched_job_id`
- `queue`

其中 `queue` 里的状态含义是：

- `queued`: 已入队，等待执行
- `running`: 正在执行
- `succeeded`: 已成功并创建 PR
- `failed`: 执行失败
- `no_change`: 执行完成但没有代码改动
- `needs_human`: 需要人工介入
- `cancelled`: 任务被取消

## 7. 怎么看 systemd 服务

看状态：

```bash
sudo systemctl status --no-pager coding-bot
```

看实时日志：

```bash
sudo journalctl -u coding-bot -f
```

常见正常日志包括：

- `polling enabled`
- `queued job ...`
- `Starting gunicorn`
- `Booting worker`

如果服务没起来，先跑：

```bash
UV_CACHE_DIR=/tmp/coding-bot-uv-cache uv run --frozen coding-bot --env-file .env doctor
```

## 8. 怎么看单个任务

每个任务都会有一个独立目录：

```text
data/jobs/<job_id>/
```

通常会看到这些文件：

- `job.json`
- `worker.log`
- `codex.stdout.log`
- `codex.stderr.log`
- `codex.last_message.txt`
- `test.stdout.log`
- `test.stderr.log`

说明：

- `job.json` 记录触发原因和原始 payload
- `worker.log` 是 worker 进程输出，可能很少，甚至为空
- `codex.stdout.log` / `codex.stderr.log` 是 Codex 执行输出
- `codex.last_message.txt` 是 Codex 最终结构化结果
- `test.*.log` 只有显式配置了 `TEST_COMMAND` 才会生成

## 9. 数据目录里都放了什么

默认数据都在项目根目录下的 `data/`：

- `data/issue_bot.db`
  SQLite 数据库，保存 Issue 和 Job 状态
- `data/jobs/`
  每个任务的日志和输出
- `data/repos/`
  本地复用的仓库缓存
- `data/active/`
  运行中的锁文件
- `data/state.json`
  轮询状态和触发去重状态

如果你改了 `.env` 里的 `DATA_DIR`、`DB_PATH`、`JOB_ROOT` 等配置，实际路径以你的配置为准。

## 10. 失败后怎么处理

先看 3 个地方：

1. `journalctl -u coding-bot -f`
2. `data/jobs/<job_id>/`
3. `curl -s http://127.0.0.1:9081/health`

最常见的失败类型包括：

- Codex 可执行文件损坏或路径错误
- GitHub App 权限不足
- fork 不存在或不可写
- 显式配置的 `TEST_COMMAND` 本身写错或不适配当前仓库
- `merge.sh` 或自定义脚本失败

如果是一次性的执行失败，最简单的重试方式通常是：

1. 确认问题已经修好
2. 在同一个 Issue 下重新评论一次新的 `/run`

## 11. 常见日常操作

重新跑 doctor：

```bash
cd /path/to/coding-bot
UV_CACHE_DIR=/tmp/coding-bot-uv-cache uv run --frozen coding-bot --env-file .env doctor
```

重启服务：

```bash
sudo systemctl restart coding-bot
```

查看最近 200 行日志：

```bash
sudo journalctl -u coding-bot -n 200 --no-pager
```

看当前健康状态：

```bash
curl -s http://127.0.0.1:9081/health
```

## 12. 使用注意事项

- 不要手工改 `data/repos/` 里的仓库缓存，尤其是在任务运行中。
- 同一个仓库是串行执行，不要把“排队慢”误判成“机器人挂了”。
- 轮询模式不是实时模式，`/run` 到被发现之间通常会有一个 `POLL_INTERVAL_SECONDS` 的延迟。
- 如果多个服务共用同一个 GitHub 机器人身份，不要假设评论作者能区分具体来源服务。
- 当前系统更偏运维型机器人，不是面向最终业务用户的即时交互工具。
