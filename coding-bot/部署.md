# Coding Bot 部署文档

## 1. 目录准备

项目可以放在任意目录，不强制 `/opt/coding-bot`。下面只用它当示例：

```bash
mkdir -p /opt
cd /opt
git clone <你的仓库地址> coding-bot
cd /opt/coding-bot
```

如果你是手工上传目录，也只要保证最后进入项目根目录即可。

## 2. 安装系统依赖

```bash
apt update
apt install -y python3 git curl gh
```

安装 `uv`：

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
export PATH="$HOME/.local/bin:$PATH"
uv --version
```

`gunicorn` 不需要单独用 apt 安装，它已经在 `pyproject.toml` 里由 `uv` 管理。

## 3. 安装并登录 Codex

确认本机能执行：

```bash
codex --help
```

然后确认你已经在本机登录过 Codex：

```bash
ls -la ~/.codex
```

## 4. 准备 GitHub App

### 4.1 建议事件

即使默认不用 webhook，也建议 App 订阅下面两个事件，后续需要时可直接切换：

- `Issues`
- `Issue comment`

### 4.2 仓库权限

安装在上游仓库或组织上的 App，至少需要：

- `Issues: Read and write`
- `Pull requests: Read and write`
- `Contents: Read and write`

如果你要让 Bot 自动创建 fork，还需要把同一个 App 安装到你的 fork 账号，并再具备：

- `Administration: Read and write`

### 4.3 下载私钥

```bash
mkdir -p /opt/coding-bot/secrets
mv <下载的私钥文件> /opt/coding-bot/secrets/github-app.pem
chmod 600 /opt/coding-bot/secrets/github-app.pem
```

## 5. 准备 Git 推送 SSH 密钥

生成密钥：

```bash
mkdir -p /opt/coding-bot/secrets
ssh-keygen -t ed25519 -f /opt/coding-bot/secrets/github-push-key -C "coding-bot"
```

查看公钥：

```bash
cat /opt/coding-bot/secrets/github-push-key.pub
```

把公钥加到你的 GitHub 账号：

- `Settings`
- `SSH and GPG keys`
- `New SSH key`

测试 SSH：

```bash
ssh -i /opt/coding-bot/secrets/github-push-key -o IdentitiesOnly=yes -T git@github.com
```

## 6. 准备 fork 仓库

你有两种方式：

### 方式 A：手工创建 fork

这是最简单的方式，直接在 GitHub 页面把上游仓库 fork 到你自己的账号。

### 方式 B：让 Bot 自动创建 fork

需要同时满足：

- GitHub App 已安装到上游仓库
- GitHub App 也安装到你的 fork 账号
- `.env` 配置了 `GITHUB_FORK_INSTALLATION_ID`

## 7. 填写 `.env`

复制模板：

```bash
cd /opt/coding-bot
cp .env.template .env
```

编辑配置：

```bash
nano /opt/coding-bot/.env
```

默认推荐保留这两个开关：

```env
ENABLE_WEBHOOK=false
ENABLE_POLLING=true
```

这是推荐的最小可迁移部署方式。

### 最小配置示例

```env
APP_HOME=.
DATA_DIR=data
SECRETS_DIR=secrets
LISTEN_HOST=0.0.0.0
LISTEN_PORT=9081

ENABLE_WEBHOOK=false
GITHUB_WEBHOOK_SECRET=
GITHUB_APP_ID=123456
GITHUB_INSTALLATION_ID=123456789
GITHUB_FORK_INSTALLATION_ID=
GITHUB_PRIVATE_KEY_PATH=secrets/github-app.pem
GITHUB_FORK_OWNER=your-github-name

ALLOWED_REPOS=upstream-owner/repo-name

RUN_ON_ISSUE_OPENED=false
TRIGGER_LABEL=ai-run
TRIGGER_COMMENT=/run

CODEX_BIN=/root/.npm-global/bin/codex
CODEX_MODEL=gpt-5.4
CODEX_SOURCE_HOME=/root/.codex
CODEX_RUNTIME_HOME=data/codex-runtime

DB_PATH=data/issue_bot.db
JOB_ROOT=data/jobs
REPO_ROOT=data/repos
ACTIVE_DIR=data/active
STATE_FILE=data/state.json

SYNC_SCRIPT_PATH=scripts/sync.sh
MERGE_SCRIPT_PATH=scripts/merge.sh

SUBMIT_COMMENT_AFTER_PR=true
SUBMIT_COMMENT_BODY=/submit

ENABLE_POLLING=true
POLL_INTERVAL_SECONDS=60
DISPATCH_INTERVAL_SECONDS=5

GITHUB_CLONE_MODE=ssh
GITHUB_CLONE_SSH_KEY_PATH=secrets/github-push-key
```

默认建议不要配置 `TEST_COMMAND`，保持留空。这样外层机器人不会再额外跑固定 shell 测试，只保留 Codex 在任务里的最小必要自检。

### 多仓库示例

多个仓库直接写成逗号分隔，不要换行：

```env
ALLOWED_REPOS=yeying-community/deployer,yeying-community/another-repo,foo/bar
```

当前实现的行为是：

- 同一个仓库内部串行处理，避免复用目录冲突
- 不同仓库之间可以分别排队和调度

### 这些项通常必须确认

- `GITHUB_APP_ID`
- `GITHUB_INSTALLATION_ID`
- `GITHUB_PRIVATE_KEY_PATH`
- `GITHUB_FORK_OWNER`
- `ALLOWED_REPOS`
- `CODEX_BIN`
- `CODEX_SOURCE_HOME`
- `GITHUB_CLONE_SSH_KEY_PATH`

## 8. 一键安装

默认推荐直接用一键脚本：

```bash
cd /opt/coding-bot
sudo BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/bootstrap.sh
```

这个脚本会按顺序执行：

- `uv sync --frozen`
- `.env` 基础检查
- `uv run --frozen coding-bot --env-file .env doctor`
- `scripts/install_systemd.sh`

如果 `.env` 不存在，它会自动从 `.env.template` 复制一份，然后提示你补齐配置后重新执行。

如果你想先只做依赖安装和自检，不安装 systemd：

```bash
cd /opt/coding-bot
INSTALL_SYSTEMD=false UV_BIN="$HOME/.local/bin/uv" ./scripts/bootstrap.sh
```

## 9. 手工执行各步骤

如果你不想走一键脚本，也可以手工拆开执行。

### 9.1 安装项目依赖

```bash
cd /opt/coding-bot
uv sync --frozen
```

### 9.2 先跑自检

```bash
cd /opt/coding-bot
UV_CACHE_DIR=/tmp/coding-bot-uv-cache uv run --frozen coding-bot --env-file .env doctor
```

如果输出包含：

```text
Doctor 完成：全部通过。
```

再继续下一步。

## 10. 本机手工启动

先直接按最终线上方式启动：

```bash
cd /opt/coding-bot
CODING_BOT_ENV_FILE=.env UV_CACHE_DIR=/tmp/coding-bot-uv-cache \
uv run --frozen gunicorn -c gunicorn.conf.py issue_bot_service:APP
```

另开一个终端检查：

```bash
curl -s http://127.0.0.1:9081/health
```

如果返回 JSON，就说明：

- gunicorn 已经起来
- Bot 配置已加载
- 后台轮询和调度线程也已经启动

停止前台运行：

```bash
Ctrl + C
```

## 11. 配置 systemd

推荐直接用一键脚本，不手工复制 unit：

```bash
cd /opt/coding-bot
sudo BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/bootstrap.sh
```

如果你只想单独重装 service，再执行：

```bash
cd /opt/coding-bot
sudo BOT_USER="$(id -un)" ./scripts/install_systemd.sh
```

如果 `uv` 不在 PATH 里：

```bash
cd /opt/coding-bot
sudo BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
```

这个脚本会生成类似下面的启动方式：

```bash
uv run --frozen gunicorn -c /opt/coding-bot/gunicorn.conf.py issue_bot_service:APP
```

并自动设置：

- `CODING_BOT_ENV_FILE=<你的 .env 路径>`
- `UV_CACHE_DIR=/tmp/coding-bot-uv-cache`

看服务状态：

```bash
sudo systemctl status --no-pager coding-bot
```

看实时日志：

```bash
sudo journalctl -u coding-bot -f
```

## 12. 第一次业务验证

在允许处理的仓库里：

1. 新建一个测试 Issue
2. 评论 `/run`
3. 观察日志：

```bash
sudo journalctl -u coding-bot -f
```

正常情况下你会看到这条链路：

- 轮询发现 `/run`
- 任务进入队列
- 拉起 worker
- 调用 Codex
- push 到 fork
- 创建 PR
- 回帖结果
- 自动评论 `/submit`

## 13. 常用排查命令

```bash
curl -s http://127.0.0.1:9081/health
```

```bash
cd /opt/coding-bot
UV_CACHE_DIR=/tmp/coding-bot-uv-cache uv run --frozen coding-bot --env-file .env doctor
```

```bash
sudo systemctl status --no-pager coding-bot
```

systemd 日志：

```bash
sudo journalctl -u coding-bot -n 200 --no-pager
```

实时日志：

```bash
sudo journalctl -u coding-bot -f
```
