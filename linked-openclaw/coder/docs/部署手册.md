# Coder Bot 部署手册

## 1. 适用范围

这份手册面向一台新的 Linux 机器，从零部署当前目录下这套 `Coder Bot + OpenClaw Gateway`。

系统分工：

- GitHub 侧负责触发、排队、仓库准备、提交分支、创建 PR
- Feishu 侧负责 handoff 讨论与 `/run` 执行确认
- OpenClaw 负责在飞书线程上下文里真正执行任务

部署目录不是固定的。下面统一用 `APP_DIR` 表示你的实际部署目录。

例如：

```bash
export APP_DIR="$HOME/linked-openclaw/coder"
```

默认端口：

- `coder-bot` HTTP 服务：`9081`
- `openclaw-gateway` 本地 gateway：`18831`

## 2. 机器前置条件

建议环境：

- Ubuntu 22.04 / 24.04
- 能使用 `sudo`
- 已安装 `systemd`
- 机器可以访问：
  - `api.github.com`
  - GitHub SSH
  - Feishu 开放平台
  - `https://router.yeying.pub`

先装基础工具：

```bash
sudo apt update
sudo apt install -y \
  git curl jq ca-certificates unzip \
  openssh-client build-essential \
  python3 python3-venv
```

## 3. 安装 `uv`

`uv` 是本项目的 Python 依赖与运行入口，`bootstrap.sh` 和 systemd 都会用到它。

官方安装方式：

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
uv --version
```

官方文档：

- <https://docs.astral.sh/uv/getting-started/installation/>

## 4. 安装 `gh`

运行主服务本身不直接依赖 `gh`，但：

- `doctor` 会检查它
- 人工排查 issue / PR 很方便
- OpenClaw 会话里常用 `gh issue view` 做上下文确认

Ubuntu 官方安装方式：

```bash
type -p curl >/dev/null || sudo apt install curl -y
sudo mkdir -p -m 755 /etc/apt/keyrings
curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg | \
  sudo tee /etc/apt/keyrings/githubcli-archive-keyring.gpg >/dev/null
sudo chmod go+r /etc/apt/keyrings/githubcli-archive-keyring.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | \
  sudo tee /etc/apt/sources.list.d/github-cli.list >/dev/null
sudo apt update
sudo apt install gh -y
gh --version
```

如果你要在这台机器上人工调试 GitHub，可额外登录：

```bash
gh auth login
```

官方文档：

- <https://github.com/cli/cli/blob/trunk/docs/install_linux.md>
- <https://docs.github.com/en/github-cli/github-cli/quickstart>

## 5. 安装 `openclaw`

本项目使用仓库内维护的 OpenClaw 配置：

- `config/openclaw.json`
- `data/openclaw/runtime/openclaw.runtime.json`

安装 CLI 后，直接使用仓库内配置即可。

推荐方式一：官方安装脚本

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --no-onboard
openclaw --version
```

推荐方式二：如果你已经自己管理 Node/npm

```bash
npm install -g openclaw@latest
openclaw --version
```

注意：

- 本仓库最终调用的是 `scripts/openclaw-local`
- OpenClaw CLI 安装完成后，无需额外执行 `openclaw onboard`

官方文档：

- <https://docs.openclaw.ai/install>

## 6. GitHub 侧准备

### 6.1 GitHub App

需要准备一个 GitHub App，并安装到目标上游仓库。

这个 App 至少要能完成这些动作：

- 读取 issue 与标签
- 读取仓库信息
- 需要时创建 fork
- 创建 PR
- 在 issue 下发评论

使用“fork 后 push，再回上游提 PR”的默认模式，还需要：

- 把这个 App 也安装到 fork 所属账号
- 在 `config/coder-bot.env` 里填写 `GITHUB_FORK_INSTALLATION_ID`

### 6.2 GitHub App 私钥

把 GitHub App 下载到的私钥放到：

```bash
mkdir -p secrets
chmod 700 secrets
```

目标文件：

```bash
secrets/github-app.pem
```

建议权限：

```bash
chmod 600 secrets/github-app.pem
```

### 6.3 Git Push SSH Key

服务发布 PR 时使用的是 SSH push，不是 `gh auth login` 里的凭据。

生成一对专用 key：

```bash
ssh-keygen -t ed25519 -f secrets/github-push-key -C "coder-bot"
chmod 600 secrets/github-push-key
chmod 644 secrets/github-push-key.pub
```

然后把公钥 `secrets/github-push-key.pub` 加到“能向 fork 仓库 push 的身份”上，常见做法二选一：

- 加到对应 GitHub 账号的 SSH keys
- 加到 fork 仓库的 deploy key，并赋写权限

## 7. Feishu / OpenClaw 侧准备

### 7.1 Feishu 应用

你需要一个能被 OpenClaw gateway 接入的飞书应用，并拿到：

- `FEISHU_APP_ID`
- `FEISHU_APP_SECRET`
- handoff 群聊 `chat_id`

如果你要做“群聊 handoff + 群线程讨论 + 线程里 `/run` 确认”，至少确认：

- 应用已被拉进目标群
- 事件订阅走长连接
- 应用具备读取群消息/群线程消息的权限

这条很关键：

- 缺少 `im:message.group_msg` 时，机器人通常能连上，但不会响应群线程讨论消息，也不会响应线程内 `/run`

### 7.2 OpenClaw 静态配置

复制模板：

```bash
cd "$APP_DIR"
cp config/openclaw.json.template config/openclaw.json
```

至少替换这些值：

- `__APP_DIR__`
- `<FEISHU_APP_ID>`
- `<FEISHU_APP_SECRET>`
- `<FEISHU_GROUP_ID>`
- `<OPENCLAW_GATEWAY_TOKEN>`

如果你打算使用群聊 handoff，不要随便改掉下面这些群线程配置：

- `channels.feishu.groupAllowFrom`
- `channels.feishu.groups.<group_id>.groupSessionScope = group_topic`
- `channels.feishu.groups.<group_id>.topicSessionMode = enabled`
- `channels.feishu.groups.<group_id>.replyInThread = enabled`

这几个字段直接影响“讨论消息是否留在同一话题框里”。

## 8. 拉代码与初始化配置

```bash
mkdir -p "$(dirname "$APP_DIR")"
git clone <your-repo-url> "$APP_DIR"
cd "$APP_DIR"
cp config/coder-bot.env.template config/coder-bot.env
```

建议先建目录：

```bash
mkdir -p data/logs data/openclaw/runtime data/openclaw/state secrets
```

## 9. 填写 `config/coder-bot.env`

这几个字段是最关键的：

### 9.1 GitHub App

```env
GITHUB_APP_ID=
GITHUB_INSTALLATION_ID=
GITHUB_FORK_INSTALLATION_ID=
GITHUB_PRIVATE_KEY_PATH=secrets/github-app.pem
GITHUB_CLONE_SSH_KEY_PATH=secrets/github-push-key
GITHUB_FORK_OWNER=your-github-name
```

说明：

- `GITHUB_INSTALLATION_ID`：上游仓库安装 ID
- `GITHUB_FORK_INSTALLATION_ID`：fork 账号安装 ID；如果没有自动 fork 需求，可先留空，但自动 fork 会失败
- `GITHUB_FORK_OWNER`：最终 push 分支的 fork 账号

### 9.2 允许处理的仓库

```env
ALLOWED_REPOS=yeying-community/deployer,yeying-community/robot
```

### 9.3 触发策略

推荐配置：

```env
ENABLE_POLLING=true
ENABLE_WEBHOOK=false
TRIGGER_LABEL=ai-run
RUN_ON_ISSUE_OPENED=false
```

说明：

- GitHub 触发只负责“发现 issue 并发 handoff”
- 真正执行仍然要在线程里发送 `/run` 或命中 `FEISHU_CONFIRM_KEYWORDS`

### 9.4 OpenClaw / 模型

```env
EXECUTION_BACKEND=openclaw
OPENCLAW_BIN=./scripts/openclaw-local
OPENCLAW_MODEL=router/gpt-5.4
OPENCLAW_TIMEOUT=1800
OPENCLAW_CONFIG_PATH=config/openclaw.json
OPENCLAW_RUNTIME_CONFIG_PATH=data/openclaw/runtime/openclaw.runtime.json
OPENCLAW_STATE_DIR=data/openclaw/state
ROUTER_API_KEY=...
```

### 9.5 Feishu handoff

```env
FEISHU_HANDOFF_CHAT_ID=oc_xxx
FEISHU_ACCOUNT_ID=default
FEISHU_CONFIRM_KEYWORDS=/run,开始执行,确认执行,可以执行
```

说明：

- `FEISHU_HANDOFF_CHAT_ID` 建议直接填 handoff 群聊的 `chat_id`
- 留空时，服务会回退到 `config/openclaw.json` 里的 `groupAllowFrom` 第一个群

## 10. 首次自检与一键安装

推荐直接跑：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/bootstrap.sh
```

`bootstrap.sh` 会做这些事：

- `uv sync --frozen`
- 检查 `config/coder-bot.env`
- 生成 `data/openclaw/runtime/openclaw.runtime.json`
- 运行 `doctor`
- 安装并重启 `openclaw-gateway.service`
- 安装并重启 `coder-bot.service`

如果你只想先做依赖安装和自检，不安装 systemd：

```bash
INSTALL_SYSTEMD=false BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/bootstrap.sh
```

## 11. systemd 部署与重启

单独重装两个服务：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
```

如果你是从旧版本升级到当前重构后的 `src/` 结构，这一步不是可选项，而是必须执行。

原因：

- 旧部署的 `coder-bot.service` 可能还在指向 `issue_bot_service:APP`
- 当前版本的正式入口已经变成 `src.main:APP`
- 如果 systemd 还跑旧入口，父进程虽然看起来能启动，但 worker 可能继续引用旧的单文件路径，导致任务真正执行时失败

重装后立刻检查：

```bash
sudo systemctl status --no-pager coder-bot
```

你应该看到类似：

```text
... gunicorn -c /your/app/config/gunicorn.conf.py src.main:APP
```

如果你仍然看到：

```text
... gunicorn ... issue_bot_service:APP
```

说明 systemd 还没切到重构后的入口，请重新执行：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
sudo systemctl daemon-reload
sudo systemctl restart openclaw-gateway coder-bot
```

常用命令：

```bash
sudo systemctl restart openclaw-gateway coder-bot
sudo systemctl stop openclaw-gateway coder-bot
sudo systemctl status --no-pager openclaw-gateway
sudo systemctl status --no-pager coder-bot
sudo journalctl -u openclaw-gateway -n 100 --no-pager
sudo journalctl -u coder-bot -n 100 --no-pager
```

## 12. 不用 systemd 的手动启动方式

### 12.1 手动启动 gateway + gunicorn

```bash
cd "$APP_DIR"
uv sync --frozen
./scripts/start_gateway.sh
CODER_BOT_ENV_FILE=config/coder-bot.env uv run --frozen gunicorn -c config/gunicorn.conf.py src.main:APP
```

### 12.2 直接用 Python 包入口启动 bot

如果你只是本地联调，也可以直接走包入口：

```bash
cd "$APP_DIR"
uv sync --frozen
python -m src --env-file config/coder-bot.env serve
```

这个入口同样会加载：

- `config/coder-bot.env`
- `config/openclaw.json`
- `data/openclaw/runtime/openclaw.runtime.json`

并启动轮询、调度线程和 Flask 服务。

查看 gateway：

```bash
./scripts/status_gateway.sh
tail -f data/logs/openclaw-gateway.log
```

查看 bot：

```bash
tail -f data/logs/gunicorn.error.log
```

## 13. 验收检查

先看基础健康：

```bash
curl -s http://127.0.0.1:9081/health
./scripts/status_gateway.sh
```

建议重点确认：

- `/health` 返回 `status=ok`
- `/health` 里的 `openclaw_config_path` 指向 `config/openclaw.json`
- `/health` 里的 `openclaw_runtime_config_path` 指向 `data/openclaw/runtime/openclaw.runtime.json`
- `./scripts/status_gateway.sh` 能看到 `agents.defaults.model.primary`
- `doctor` 能通过 `GitHub Installation Token` 和 `Fork Installation Token`

看某个 issue 的会话：

```bash
curl -s http://127.0.0.1:9081/issues/<owner>/<repo>/<issue_number>/session
```

## 14. 建议的联调流程

1. 新建一个测试 issue，或给现有 issue 打上 `ai-run`
2. 观察机器人是否在飞书 handoff 群里创建讨论线程
3. 先在线程里讨论，不要直接在 issue 评论区触发
4. 在线程里发送 `/run`
5. 观察是否进入执行
6. 成功时应在同一飞书线程回“已完成 + PR 链接”
7. 失败时应在同一飞书线程回“执行失败 + 错误摘要”

## 15. 常见问题

### 15.1 `uv not found`

说明 `UV_BIN` 不对，或 `~/.local/bin` 没进 PATH。先执行：

```bash
echo "$PATH"
ls -l "$HOME/.local/bin/uv"
```

### 15.2 `openclaw not found`

说明 OpenClaw CLI 没装好，或者不在 PATH。先执行：

```bash
command -v openclaw
openclaw --version
```

### 15.3 `router/* but ROUTER_API_KEY is empty`

说明你选的是 `router/gpt-5.4`，但没填 `ROUTER_API_KEY`。

### 15.4 飞书群线程里没反应

优先检查：

- 飞书应用是否在目标群里
- 事件订阅是否已切到长连接
- 是否开通了 `im:message.group_msg`
- `config/openclaw.json` 里的群 ID 是否和实际 handoff 群一致
- `replyInThread` / `topicSessionMode` 是否被改坏

### 15.5 能建线程，但 `/run` 不执行

优先检查：

- `FEISHU_CONFIRM_KEYWORDS` 是否包含 `/run`
- `/run` 是否就是发在 handoff 线程里，而不是群主聊天面板
- `curl -s http://127.0.0.1:9081/health` 里的 `last_poll_error`
- `tail -f data/logs/gunicorn.error.log`

### 15.6 服务已经升级代码，但实际仍跑旧入口

典型现象：

- `/health` 看起来正常
- 讨论线程和 `/run` 都能走通
- 真正执行时 worker 日志里还在引用旧文件，例如 `issue_bot_service.py`

优先检查：

```bash
sudo systemctl status --no-pager coder-bot
```

确认主命令是否为：

```text
src.main:APP
```

如果不是，重新安装并重启：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" UV_BIN="$HOME/.local/bin/uv" ./scripts/install_systemd.sh
sudo systemctl restart openclaw-gateway coder-bot
```

### 15.7 PR 创建失败或 push 失败

优先检查：

- `GITHUB_FORK_OWNER` 是否正确
- `GITHUB_FORK_INSTALLATION_ID` 是否正确
- `secrets/github-push-key` 是否真能向 fork push
- fork 是否已经存在

## 16. 常用命令汇总

```bash
cd "$APP_DIR"
uv sync --frozen
UV_CACHE_DIR=/tmp/coder-bot-uv-cache uv run --frozen coder-bot --env-file config/coder-bot.env doctor
./scripts/start_gateway.sh
./scripts/status_gateway.sh
curl -s http://127.0.0.1:9081/health
curl -s http://127.0.0.1:9081/issues/<owner>/<repo>/<issue_number>/session
sudo systemctl restart openclaw-gateway coder-bot
sudo systemctl status --no-pager coder-bot
```
