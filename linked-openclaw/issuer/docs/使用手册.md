# Issuer 使用手册

## 1. 标准链路

系统执行链路：

1. 飞书侧 `@issuer` 提出 create / update / close / comment 请求
2. OpenClaw 在当前群会话里理解意图并生成草案
3. 草案写入 `data/openclaw/workspace-larkbot/state/pending-actions.sqlite3`
4. 用户发送 `/confirm`、`/confirm robot`、`/confirm draft:xxxx`
5. `confirmation-bridge` hook 校验权限后执行真正的 GitHub API 调用
6. 执行结果回到同一个飞书会话

默认规则：

- 不直接执行破坏性动作
- 先出草案，再确认
- 默认只允许原发起人或 `config/policy.json` 里的管理员确认
- 同群同用户同仓库只保留 1 个草案
- 不同仓库可以并行
- 多草案时推荐显式写仓库名

## 2. 当前启动方式

手动启动：

```bash
cd "$APP_DIR"
./scripts/start_gateway.sh
./scripts/status_gateway.sh
```

如果你改了 `workspace_assets/`：

```bash
cd "$APP_DIR"
./scripts/sync_workspace.sh
./scripts/stop_gateway.sh
./scripts/start_gateway.sh
```

systemd 方式：

```bash
cd "$APP_DIR"
BOT_USER="$(id -un)" ./scripts/install_systemd.sh
sudo systemctl restart issuer-openclaw-gateway
```

内部入口说明：

- `scripts/start_gateway.sh`：给人手工执行
- `scripts/run_gateway.sh`：给后台进程和 systemd 执行
- `scripts/openclaw-local`：实例级 OpenClaw wrapper

## 3. 触发方式

默认支持 4 类 GitHub 动作：

- 创建 issue
- 更新 issue
- 关闭 issue
- 对明确指定的 issue 追加评论

飞书里常见表达：

- 创建：`@issuer 在 robot 创建 issue，标题是 xxx，内容是 yyy`
- 更新：`@issuer 修改 robot #123 的 title / body / labels / assignees`
- 关闭：`@issuer 关闭 robot #123，原因 completed`
- 评论：`@issuer 给 https://github.com/yeying-community/robot/issues/123 评论 /run`

帮助入口：

```text
@issuer /help
```

## 4. 草案模型

每个待确认动作都会落成一条草案记录。

当前草案的稳定定位维度是：

- 群会话
- 发起人
- 仓库

这意味着：

- 同一个人对同一个仓库重复发起时，新草案会覆盖旧草案
- 同一个人对不同仓库可以同时挂多个草案
- 不同人互不影响

确认方式：

- 单草案：`/confirm`、`/submit`
- 多草案：`/confirm robot`
- 精确草案：`/confirm draft:5d496c27`

取消方式同理：

- `/cancel`
- `/cancel robot`
- `/cancel draft:5d496c27`

## 5. 平时怎么查状态

先看 gateway：

```bash
./scripts/status_gateway.sh
tail -f data/logs/openclaw-gateway.log
```

再看 issuer 审计：

```bash
tail -f data/logs/issuer-audit.jsonl
```

看当前 pending：

```bash
./scripts/inspect_pending.sh summary
./scripts/inspect_pending.sh list
```

按群查：

```bash
./scripts/inspect_pending.sh conversation --conversation-id chat:oc_xxx
```

按用户查：

```bash
./scripts/inspect_pending.sh requester --requester-id ou_xxx
```

按 draftId 反查详情：

```bash
./scripts/inspect_pending.sh show --draft-id 5d496c27
```

## 6. OpenClaw 与 GitHub 的分工

- `workspace_assets/tools/github_app_token.mjs` 负责 GitHub App 认证
- `workspace_assets/tools/github_issue_create.mjs` 负责 create preview / execute
- `workspace_assets/tools/github_issue_update.mjs` 负责 update preview / execute
- `workspace_assets/tools/github_issue_close.mjs` 负责 close preview / execute
- `workspace_assets/tools/github_issue_comment.mjs` 负责明确 issue comment 的直接执行
- `workspace_assets/tools/pending_action.mjs` 负责草案保存、读取、执行、清理
- `workspace_assets/hooks/confirmation-bridge/handler.ts` 负责 `/help`、`/confirm`、`/cancel`
- OpenClaw gateway 负责 Feishu 接入、会话、模型调用和 hook 装载

配置和状态目录也要分开理解：

- `config/openclaw.json`：你手工维护的静态配置
- `config/github-app.config.env`：GitHub App / token 配置
- `data/openclaw/workspace-larkbot/`：运行时真正执行的 workspace
- `data/openclaw/state/`：OpenClaw 自己的状态目录

## 8. 常见排查

如果飞书里完全没反应，优先检查：

- `config/openclaw.json` 里的 `channels.feishu.appId` / `appSecret`
- `channels.feishu.groupAllowFrom` 是否包含当前群 `chat_id`
- 当前消息是否真的 `@issuer`
- `./scripts/status_gateway.sh` 的最近日志

如果有处理但群里没回消息，优先检查：

- `config/openclaw.json` 里的 `messages.groupChat.visibleReplies`
- 当前是否显式用了 `automatic`
- 日志里是否出现 `dispatch complete (... replies=0)`

如果整体反应很慢，优先检查：

- 上游模型是否超时
- 日志里是否有 `llm-idle-timeout`
- 日志里是否有 `Concurrency limit exceeded for account`
- 日志里是否有 `internal_server_error`、`unexpected EOF`

如果 `/confirm` 没生效，优先检查：

- 发 `/confirm` 的人是不是原发起人
- 或者这个人是否在 `config/policy.json` 的 `admins` 里
- 当前会话里是否确实存在 pending action

如果你改了源码但效果没变，优先检查：

- 是否已经执行 `./scripts/sync_workspace.sh`
- 是否已经重启 gateway
- 你改的是 `workspace_assets/` 还是 `data/openclaw/workspace-larkbot/`

## 9. 配置读取说明

`github_app.mjs` 不是自动优先读系统全局环境，而是按这个顺序读：

1. `GITHUB_APP_ENV_FILE` 或 `GITHUB_ENV_FILE`
2. 从运行时 workspace 向上找应用根目录
3. 默认读取 `APP_DIR/config/github-app.config.env`

所以当前这套实现是目录内配置，不再依赖旧的用户级 OpenClaw 配置目录。
